# Bevy 简述（Bevy TLDR）

Bevy 是一个基于 Rust 的原型式（archetypal）实体组件系统（ECS）游戏引擎。
它强调模块化、性能和易用性。

<details markdown="1">
<summary>原文</summary>

Bevy is an archetypal Entity Component System (ECS) game engine built in Rust.
It emphasizes modularity, performance, and ease of use.

</details>

## 实体与组件（Entity and Components）

`Entity` 只是一个用于查找关联 `Component` 的标识符。`Component` 才是真正存储构成实体数据的地方。

<details markdown="1">
<summary>原文</summary>

An `Entity` is just an identifier to find associated `Component`.
A `Component` is what actually stores the data that makes up each entity.

</details>

一个很好的心智模型是：实体代表内存数据库中的"行"，而组件代表"列"。

<details markdown="1">
<summary>原文</summary>

A good mental model to use is that entities represent a row in an in-memory database, while components are our columns.

</details>

每个 `Entity` 对于同一种类型的 `Component` 只能拥有一个实例。这些组件可以在实体生命周期中动态添加和移除。
所有内容都存储在 `World` 中，而 `World` 由 `App` 管理。

<details markdown="1">
<summary>原文</summary>

Each `Entity` can only have a single `Component` per type.
These components can be added and removed dynamically over the course of the entity's lifetime.
Everything is stored inside a `World` which is managed by an `App`.

</details>

我们通过派生 `Component` trait 来定义组件：

<details markdown="1">
<summary>原文</summary>

We define components by deriving the `Component` trait:

</details>

```rust
#[derive(Component)]
struct Player;

#[derive(Component)]
enum Ship {
    Destroyer,
    Cruiser,
    Battleship,
}

#[derive(Component)]
struct Health(f32);

#[derive(Component)]
#[component(on_add = on_position_added)]
struct Position {
    x: i32,
    y: i32,
}
```

组件有 5 种可用于处理"必须发生"的副作用的生命周期钩子：

<details markdown="1">
<summary>原文</summary>

Components have 5 different life-cycle hooks we can use to handle side effects that __need__ to happen:

</details>

1. `#[component(on_add = on_add_function)]`
2. `#[component(on_insert = on_insert_function)]`
3. `#[component(on_replace = on_replace_function)]`
4. `#[component(on_remove = on_remove_function)]`
5. `#[component(on_despawn = on_despawn_function)]`

传给这些钩子函数的系统参数是 `DeferredWorld` 和 `HookContext`。

<details markdown="1">
<summary>原文</summary>

The systems we pass to these take a `DeferredWorld` and a `HookContext`.

</details>

```rust
#[derive(Component, Default)]
#[component(on_add = on_ship_added)]
enum Ship {
    Destroyer,
    Frigate,
    #[default]
    Scout,
}

fn on_ship_added(mut world: DeferredWorld, context: HookContext) {
    world
        .commands()
        .entity(context.entity)
        .insert(Position { x: 0, y: 0 });
}
```

要添加依赖于其他组件的组件，首选方式是把它们声明为必需的（require）：

<details markdown="1">
<summary>原文</summary>

The preferred way to add components based on other components is to make them required:

</details>

```rust
#[derive(Component)]
#[require(Position, Ship)]
struct Player;

fn spawn_player_with_required_components(
    mut commands: Commands
) {
    commands.spawn(Player);
}
```

当一个组件被生成时，如果它声明了必需组件，那这些组件都会被自动补齐，除非我们显式覆盖。
唯一要求是每个必需组件都实现 `Default` 特征。

<details markdown="1">
<summary>原文</summary>

When a component is spawned, if it has any required components,
it will automatically add them __unless__ we override them.
The only requirement is that each required component implements the `Default` trait.

</details>

这些 required 调用是 **递归** 的：如果你 required 的组件本身也 required 了其他组件，它们也会一起被添加。

<details markdown="1">
<summary>原文</summary>

All these required calls are __recursive__.
If a component you require has required components, they will also be added.

</details>

`Resource` 是一种没有任何 `Entity` 的特殊组件。它们是单例组件，在我们系统里查询通常更方便。

<details markdown="1">
<summary>原文</summary>

A `Resource` is a special kind of component that has no `Entity`.
These are singleton components that are easier to query for inside our systems.

</details>

```rust
#[derive(Resource)]
struct Score(usize);

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .init_resource::<Score>()
        .run();
}
```

## 系统（Systems）

系统是我们触发副作用、改变游戏状态的地方。

<details markdown="1">
<summary>原文</summary>

Systems are where we trigger side effects that change our game's state.

</details>

在 Bevy 中，系统就是普通 Rust 函数，但有一条规则：它们的参数必须实现 `SystemParam`。

<details markdown="1">
<summary>原文</summary>

In Bevy, systems are simple rust functions with one rule: They can only have parameters that implement `SystemParam`.

</details>

```rust
fn spawn_player(mut commands: Commands) {
    // Spawns a single entity with multiple components
    commands.spawn((
        Player,
        Ship::Destroyer,
        Health(100.0),
        Position { x: 1, y: 2 }
    ));
}
```

`Commands` 用于修改 `World` 的状态，这比让每个系统直接可变地访问 `World` 更高效。

<details markdown="1">
<summary>原文</summary>

`Commands` are what we use to change the state of our `World` in a way that is
more performant than letting each system mutate the world directly.

</details>

当你使用 `Commands` 作为系统参数时，实际上是在把命令排入 `CommandQueue`，并在切换到由 `App` 推进的下一个 `Schedule` 时执行。

<details markdown="1">
<summary>原文</summary>

When you use the system parameter `Commands` you are enqueuing your commands to the `CommandQueue`
which runs when we transition to the next `Schedule` which is advanced by the `App`.

</details>

## 应用（Apps）

所有内容都是通过 `App` 协调，`App` 负责把系统安排在游戏循环中的特定时机运行：

<details markdown="1">
<summary>原文</summary>

Everything is coordinated through an `App` which schedules our systems to run at
certain points in the game's loop:

</details>

```rust
use bevy::prelude::*;

fn main() {
    App::new()
        .add_systems(Startup, setup_everything)
        .add_systems(Update, process_input)
        .add_systems(FixedUpdate, move_player)
        .run();
}
```

你通常会把逻辑放进三个主要调度标签：

1. `Update` - 每个循环运行一次
2. `FixedUpdate` - 每固定时间运行一次
3. `Startup` - 启动时只运行一次

<details markdown="1">
<summary>原文</summary>

You will mostly be adding your logic to the three main schedule labels:

1. `Update` runs once every loop
2. `FixedUpdate` runs once every fixed amount of time
3. `Startup` runs once at startup

</details>

此外还有其他内置调度标签可用于更细粒度的时机控制：

1. `PreStartup`
2. `Startup`
3. `PostStartup`
4. `First`
5. `PreUpdate`
6. `StateTransition`
7. `RunFixedUpdateLoop` - 条件性运行 `FixedUpdate`
8. `Update`
9. `PostUpdate`
10. `Last`

<details markdown="1">
<summary>原文</summary>

Additionally there are other built-in schedule labels for more specific use:

1. `PreStartup`
2. `Startup`
3. `PostStartup`
4. `First`
5. `PreUpdate`
6. `StateTransition`
7. `RunFixedUpdateLoop` which runs `FixedUpdate` conditionally
8. `Update`
9. `PostUpdate`
10. `Last`

</details>

这些类型每个都是一个 `ScheduleLabel`。标签是用来标识某个 `Schedule`；而 `Schedule` 包含在特定条件下执行系统所需的元数据和执行器。

<details markdown="1">
<summary>原文</summary>

These types are each a `ScheduleLabel`. Labels are used to identify a `Schedule`
which contains the metadata and executor needed to run them under certain conditions.

</details>

只要不存在可变数据访问冲突，Bevy 就会尽可能并行运行所有系统。原型（Archetype）用于优化这个过程的性能。

<details markdown="1">
<summary>原文</summary>

Bevy will try and run all systems in parallel as long as there are no mutable data access conflicts.
Archetypes are used as a performance optimization for this process.

</details>

可以向 `App` 提供一个状态枚举，用于管理不同操作模式：

<details markdown="1">
<summary>原文</summary>

An `App` can be given a state enum to manage different modes of operation:

</details>

```rust
#[derive(Debug, Clone, Eq, PartialEq, Hash, Default, States)]
enum AppState {
    #[default]
    MainMenu,
    InGame,
    Paused,
}

fn main() {
    App::new()
        // Add our state to our app definition
        .init_state::<AppState>()
        // We can add systems to trigger during transitions
        .add_systems(OnEnter(AppState::MainMenu), spawn_menu)
        // Or we can use run conditions
        .add_systems(Update, play_game.run_if(in_state(AppState::InGame)))
        .run();
}
```

如果我们想实现显式状态流转，可以把逻辑写在状态类型本身上：

<details markdown="1">
<summary>原文</summary>

If we wanted to create explicit transitions we could implement the logic on our state:

</details>

```rust
impl AppState {
    fn next(&self) -> Self {
        match *self {
            AppState::MainMenu => AppState::InGame,
            AppState::InGame => AppState::Paused,
            AppState::Paused => AppState::InGame,
        }
    }
}
```

插件是把相关功能分组组织起来的一种方式。

<details markdown="1">

Plugins are a way to group related functionality together.

<summary>原文</summary>

</details>

```rust
fn plugin(app: &mut App) {
```

<details markdown="1">

```rust
pub struct CameraPlugin;

impl Plugin for CameraPlugin {

    }

fn initialize_camera(mut commands: Commands) {
    commands.spawn(Camera2d);
}
```

## 查询（Querying）

为了在系统中访问实体的组件，可以使用 `Query<D, F>` 系统参数：

<details markdown="1">
<summary>原文</summary>

To access the components of an entity inside our systems we can use the `Query<D, F>` system parameter:

</details>

```rust
fn fetch_players(query: Query<&Player>) {
    for player in &query {
        info!("Player: {:?}", player);
    }
}
```

`Query` 允许我们通过两个泛型参数来指明我们在每个实体中想要的数据：

<details markdown="1">
<summary>原文</summary>

The `Query` system parameter lets us specify the data we want from each entity using the two generic parameters:

</details>

```rust
//     ------- the `QueryData`
//    |  ---- the `QueryFilter`
//    v  v
Query<D, F>

//      --------- Give us read-only access to all the `Transform` components
//     |          ---- Which have a `Player` component on the same entity
//     v          v
Query<&Transform, With<Player>>

//                     --- NOTE: Each parameter can be a tuple as well
//                    |
//                    v
Query<&mut Transform, (With<Player>, With<Living>)>
```

当其中一个泛型参数是元组时，该元组中的所有类型都必须被查询满足。

<details markdown="1">
<summary>原文</summary>

When one of the generic parameters is a tuple then ___all___ the types in that
tuple must be satisfied by that query.

</details>

有一些便捷类型可以更容易地表达复杂查询：

| 参数          | 描述                       |
|-------------|--------------------------|
| `Option<T>` | 一个仅当它存在时的组件，否则为 `None`   |
| `AnyOf<T>`  | 获取具有类型 T 中任何组件的实体        |
| `Ref<T>`    | 实体组件 `T` 的共享借用，可访问变化检测   |
| `Has<T>`    | 返回一个 bool，描述实体是否具有组件 `T` |
| `Entity`    | 返回实体                     |

<details markdown="1">
<summary>原文</summary>

There are convenient types that make expressing more complicated queries easier:

| parameter   | description                                                                |
|-------------|----------------------------------------------------------------------------|
| `Option<T>` | a component but only if it exists, otherwise `None`                        |
| `AnyOf<T>`  | fetches entities with any of the components in type T                      |
| `Ref<T>`    | shared borrow of an entity's component `T` with access to change detection |
| `Has<T>`    | Returns a bool that describes if an entity has the component `T`           |
| `Entity`    | returns the entity                                                         |

</details>

除了 `Query`，还有其他一些同类系统参数也执行查询：

| 系统参数                   | 描述                      |
|------------------------|-------------------------|
| `Single<D, F>`         | 精确匹配一个查询项。如果更多或没有，跳过系统。 |
| `Option<Single<D, F>>` | 匹配零个或一个查询项。如果更多，跳过系统。   |
| `Populated<D, F>`      | 匹配至少一个或更多。如果没有，跳过系统。    |

<details markdown="1">
<summary>原文</summary>

In addition to the `Query` system parameter there are other sibling system parameters
that also perform queries:

| System parameter       | Description                                                       |
|------------------------|-------------------------------------------------------------------|
| `Single<D, F>`         | Matches exactly one query item. Skips the system if more or none. |
| `Option<Single<D, F>>` | Matches zero or one query item. Skips the system if more.         |
| `Populated<D, F>`      | matches at least one or more. Skips the system if none.           |

</details>

当你确定只会有一个实体具有某类组件时，`Single` 有助于减少样板代码：

<details markdown="1">
<summary>原文</summary>

`Single` can be useful to reduce boilerplate when you know there is only ever
a single entity with a particular component:

</details>

```rust
fn move_the_only_player(mut transform: Single<&mut Transform, With<Player>>) {
    transform.translation.x += 1.
}
```

只有存在单个实体匹配我们查询的组件时，这个系统才会运行。

<details markdown="1">
<summary>原文</summary>

This system would NOT be run unless there was a single entity that matched the components we query.

</details>

`Query<D, F>` 的第二个参数是 `QueryFilter`。这些过滤器由条件类型封装：

| 方法           | 描述                        |
|--------------|---------------------------|
| `With<T>`    | 仅具有 `T` 组件的项目             |
| `Without<T>` | 仅不具有 `T` 组件的项目            |
| `Or<F>`      | 检查元组 `F` 中所有过滤器是否（任一存在）适用 |
| `Changed<T>` | 仅类型为 `T` 且在此 tick 中更改的组件  |
| `Added<T>`   | 仅在此 tick 中添加的类型为 `T` 的组件  |

<details markdown="1">
<summary>原文</summary>

The second argument in your `Query<D, F>` is the `QueryFilter`. These filters are wrapped by a condition type:

| method       | description                                             |
|--------------|---------------------------------------------------------|
| `With<T>`    | only items with a `T` component                         |
| `Without<T>` | only items without a `T` component                      |
| `Or<F>`      | checks if all filters in the tuple `F` apply            |
| `Changed<T>` | only components of type `T` that were changed this tick |
| `Added<T>`   | only components of type `T` that were added this tick   |

</details>

为了从 ECS 存储中获取组件，`Query` 系统参数提供了如下 API：

| 方法                  | 描述                             |
|---------------------|--------------------------------|
| `iter`              | 返回包含所有项的迭代器                    |
| `for_each`          | 为每个项并行运行给定函数                   |
| `iter_many`         | 为匹配实体列表的每个项运行给定函数              |
| `iter_combinations` | 返回包含指定数量项所有组合的迭代器              |
| `par_iter`          | 返回并行迭代器                        |
| `get`               | 返回给定实体的查询项                     |
| `get_component<T>`  | 返回给定实体的组件                      |
| `many`              | 返回给定实体列表的查询项                   |
| `get_single`        | 返回 `Result<T>` 的 `single` 安全版本 |
| `single`            | 返回查询项，如果还有其他的就 panic           |
| `is_empty`          | 如果查询为空，则返回 true                |
| `contains`          | 如果查询包含给定实体，则返回 true            |

<details markdown="1">
<summary>原文</summary>

To retrieve components from our ECS storage our `Query` system parameter provides an API with the following methods:

| method              | description                                                              |
|---------------------|--------------------------------------------------------------------------|
| `iter`              | returns an iterator over all items                                       |
| `for_each`          | runs the given function in parallel for each item                        |
| `iter_many`         | runs a given function for each item matching a list of entities          |
| `iter_combinations` | returns an iterator over all combinations of a specified number of items |
| `par_iter`          | returns a parallel iterator                                              |
| `get`               | returns a query item for a given entity                                  |
| `get_component<T>`  | returns the component for a given entity                                 |
| `many`              | returns a query item for a given list of entities                        |
| `get_single`        | the safe version of `single` which returns a `Result<T>`                 |
| `single`            | returns the query item while panicking if there are others               |
| `is_empty`          | returns true if the query is empty                                       |
| `contains`          | returns true if query contains a given entity                            |

</details>

每个方法也都有对应的 `*_mut` 版本，会返回可变访问，从而允许我们修改他们的数据而不只是读取。

<details markdown="1">
<summary>原文</summary>

Each method also has a corresponding `*_mut` variant which will return the components with mutable ownership.
This lets us change their data, instead of just reading it.

</details>

在已经持有某个 `Entity`（本质上是一个 ID）的情况下，我们可以使用 `get` 或 `get_mut`。

<details markdown="1">
<summary>原文</summary>

In situations where we have a particular `Entity` (which is basically an ID), we can use `get` or `get_mut`.

</details>

```rust
#[derive(Resource)]
struct PlayerRef(Entity);

fn move_player_by_component(
    mut query: Query<&mut Transform>,
    player: Res<PlayerRef>
) {
    if let Ok(mut transform) = query.get_mut(player.0) {
        transform.translation.x += 1.;
    }
}
```

而在有一组 `Entity` 并且希望仅遍历这些实体组件的情况下，我们可使用 `iter_many`。

<details markdown="1">
<summary>原文</summary>

In cases where we have a list of `Entity` and we want to iterate over only those entity components
we can use `iter_many`.

</details>

```rust
#[derive(Component)]
struct Health(pub f32);

#[derive(Resource)]
struct Selection {
    enemies: Vec<Entity>
}

const ATTACK_DAMAGE: f32 = 10.;

fn attack_selected_enemies(
    mut query: Query<&mut Health>,
    selected: Res<Selection>
) {
    let mut iter = query.iter_many_mut(&selected.enemies);
    while let Some(mut health) = iter.fetch_next() {
        health.0 -= ATTACK_DAMAGE;
    }
}
```

## 资产/资源（Assets）

加载资源使用 `AssetServer`，它负责从某个 `AssetSource`（通常是文件系统）异步加载资源。

<details markdown="1">
<summary>原文</summary>

To load assets we use the `AssetServer` which manages asynchronous loading assets
from a particular `AssetSource`, usually the filesystem.

</details>

所有资源大体遵循同一流程：

1. 如果是自定义资源，先注册 `Asset<T>` 类型
2. 如果是自定义资源，注册该资源的 `AssetLoader`
3. 将资源放入 `assets` 文件夹
4. 调用 `AssetServer::load` 获取 对于资源的 `Handle<T>`

<details markdown="1">
<summary>原文</summary>

All assets follow the same general process:

1. We register a new `Asset<T>` type if its custom
2. We Register an `AssetLoader` for that asset if its custom
3. We add the asset to our `assets` folder
4. Then we call `AssetServer::load` to get a `Handle<T>` to the asset

</details>

```rust
fn load_images(asset_server: Res<AssetServer>, mut commands: Commands) {
    // This will not block, the asset will be loaded in the background
    let image_handle: Handle<Image> = asset_server.load("images/bevy.png");

    commands.spawn(Sprite {
        image: image_handle,
        ..default()
    });
}
```

默认情况下，Bevy 会从应用根目录下的 `assets` 文件夹查找资源，这个根路径由 `BEVY_ASSET_ROOT` 环境变量控制。

<details markdown="1">
<summary>原文</summary>

By default it will expect our assets to be inside the `assets` folder inside the root directory
of our application controlled by the `BEVY_ASSET_ROOT` environment variable.

</details>

资源状态可通过两种方式追踪：

1. 通过事件（如 `AssetEvent::LoadedWithDependencies`）
2. 或者通过 `AssetServer::get_load_state` 主动查询

<details markdown="1">
<summary>原文</summary>

Assets can be tracked one of two ways:

1. Through events like `AssetEvent::LoadedWithDependencies`
2. Or by querying the asset server with `AssetServer::get_load_state`.

</details>

## 消息与事件（Messages and Events）

Bevy 中有两类事件体系：

1. `Message`：用于系统之间通信
2. `Event` / `EntityEvent`：用于由**观察器**触发的即时行为

<details markdown="1">
<summary>原文</summary>

There are two kind of events in Bevy:

1. `Message` for communication between systems
2. `Event` and `EntityEvent` for __observers__ that trigger immediate behavior

</details>

`Messages<T>` 是一个充当双缓冲队列的集合。这样可以确保每个系统都有机会看到每条消息，让系统不必过度依赖单帧内的精确执行顺序。

<details markdown="1">
<summary>原文</summary>

`Messages<T>` is a collection that acts as a double buffered queue.
This is done to ensure each system has an opportunity to see each message.
It is helping systems not have to care about the exact ordering within a frame.

</details>

消息通过派生 `Message` 特征定义，并需要注册到 `App`：

<details markdown="1">
<summary>原文</summary>

Messages are defined by deriving the `Message` trait and need to be registered to the `App`:

</details>

```rust
// With a marker message
#[derive(Message)]
struct PlayerKilled;

// With a unit type
#[derive(Message)]
struct PlayerDetected(Entity);

// With fields
#[derive(Message)]
struct PlayerDamaged {
    entity: Entity,
    damage: f32,
}

fn main() {
    App::new()
        .add_message::<PlayerKilled>()
        .add_message::<PlayerDetected>()
        .add_message::<PlayerDamaged>();
}
```

如果消息在两帧之内没有被消费，它们会被清理并静默丢弃。

<details markdown="1">
<summary>原文</summary>

If your messages are not consumed by 2 frames from now then they will be cleaned up and dropped silently.

</details>

向消息流写入时使用 `MessageWriter<T>`：

<details markdown="1">
<summary>原文</summary>

To write messages to a stream we use a `MessageWriter<T>`:

</details>

```rust
fn detect_player(
    mut messages: MessageWriter<PlayerDetected>,
    players: Query<(Entity, &Transform), With<Player>>,
) {
    for (entity, transform) in players {
        messages.write(PlayerDetected(entity));
    }
}
```

从系统读取消息时使用 `MessageReader<T>`，它会消费缓冲区中的消息：

<details markdown="1">
<summary>原文</summary>

We can read messages from our systems with an `MessageReader<T>` that consumes messages from our buffers:

</details>

```rust
fn react_to_detection(mut messages: MessageReader<PlayerDetected>) {
    for message in messages.read() {
        // Do something with each event here
    }
}
```

事件是消息的即时版本，分为两种：

1. `Event`：以 `GlobalTrigger` 定义的全局事件
2. `EntityEvent`：以 `EntityTrigger` 定义的实体特定事件

<details markdown="1">
<summary>原文</summary>

Events are the immediate version of messages. They come in two types:

1. `Event` for global events defined with a `GlobalTrigger`
2. `EntityEvent` for entity specific events defined with an `EntityTrigger`

</details>

`EntityEvent` 默认不传播。若要自动传播，事件需要用 `#[entity_event(propagate, auto_propagate)]` 标记，这会让派生宏加入传播功能。

<details markdown="1">
<summary>原文</summary>

The default behavior of `EntityEvent` is not to propagate.
To auto propagate, the events needs to be marked with `#[entity_event(propagate, auto_propagate)]`
which will tell the derive macro to add the propagation functionality.

</details>

若要手动传播，可调用 `On<T, B>::propagate(true)`。

<details markdown="1">
<summary>原文</summary>

To propagate manually you would `On<T, B>::propagate(true)`.

</details>

这些事件都由 `Observer` 消费。这是一个回调系统，并且**必须**持有 `On` 系统参数作为其**第一个**参数：

<details markdown="1">
<summary>原文</summary>

These events are consumed by an `Observer` which is a callback system that
__must__ take an `On` system parameter as the __first__ argument:

</details>

```rust
fn on_respawn(
    event: On<Add, Enemy>,
    query: Query<(&Enemy, &Position)>,
) {
    let (enemy, position) = query.get(event.entity).unwrap();
    println!("Enemy was respawned at {:?}", position);
}
```

观察器既可以是全局的（注册到 `App`）：

<details markdown="1">
<summary>原文</summary>

Observers can be global by adding them to the `App` definition:

</details>

```rust
fn main() {
    App::new().add_plugins(DefaultPlugins).add_observer(on_respawn);
}
```

也可以是局部的（只对特定实体触发）：

<details markdown="1">
<summary>原文</summary>

Or they can be local and only triggered for particular entities:

</details>

```rust
fn spawn_boss(mut commands: Commands) {
    let entity = commands.spawn((Enemy, Boss)).observe(on_boss_spawned).id();

    // Later, or potentially in another system
    commands.trigger(BossSpawned { entity });
}
```

这些实体事件会沿着 `ChildOf` 层级向上冒泡，具体取决于你是否启用了自动传播。

<details markdown="1">
<summary>原文</summary>

These entity events will bubble up a hierarchy of `ChildOf` attached components
depending on if you have set them to auto propagate or not.

</details>

下表总结了事件与消息的差异：

|            | 事件             | 消息        |
|------------|----------------|-----------|
| **最优事件频率** | 不频繁            | 频繁        |
| **处理器**    | 仅处理单个事件        | 一次可处理多条消息 |
| **延迟**     | 即时             | 最多 1 帧    |
| **事件传播**   | 冒泡             | 无         |
| **作用域**    | World 或 Entity | World     |
| **顺序**     | 无明确顺序          | 有序        |
| **耦合度**    | 高              | 低         |

<details markdown="1">
<summary>原文</summary>

This table summarizes the differences between events and messages:

|                             | Events                      | Messages                          |
|-----------------------------|-----------------------------|-----------------------------------|
| __Optimal event frequency__ | Infrequent                  | Frequent                          |
| __Handler__                 | Only handles a single event | Can handle many messages together |
| __Latency__                 | Immediate                   | Up to 1 frame                     |
| __Event propagation__       | Bubbling                    | None                              |
| __Scope__                   | World ___or___ Entity       | World                             |
| __Ordering__                | No explicit order           | Ordered                           |
| __Coupling__                | High                        | Low                               |

</details>

## 关系（Relationships）

Bevy 提供了内置的父子关系模型，由两个组件构成：

1. `ChildOf`：我们附加在其他实体上的 `Relationship`
2. `Children`：保持同步的 `RelationshipTarget`

<details markdown="1">
<summary>原文</summary>

Bevy has a built-in relationship it provides for parent/child relationships that
is made up of two components:

1. `ChildOf`: The `Relationship` we attach to other entities
2. `Children`: The `RelationshipTarget` that is kept in sync

</details>

它们会把父实体的 `Transform` 与 `GlobalTransform` 传播到子实体，以保持同步。

<details markdown="1">
<summary>原文</summary>

These will propagate `Transform` and `GlobalTransform` of the parent to its children to keep them in sync.

</details>

当你销毁父实体（持有 `Children` 的实体）时，所有 `ChildOf` 组件会自动移除。

<details markdown="1">
<summary>原文</summary>

When you despawn the parent (the entity holding the `Children`) then
all the `ChildOf` components are removed automatically.

</details>

```rust
fn spawn_ship(mut commands: Commands) {
    let fleet = commands.spawn(Fleet).id();

    commands.spawn(Ship, ChildOf(fleet));
}
```

可以通过 `with_children` 从父实体生成子实体：

<details markdown="1">
<summary>原文</summary>

We can spawn children from a parent with the `with_children` method:

</details>

```rust
fn spawn_fleet(mut commands: Commands) {
    commands
        .spawn(Fleet)
        .with_children(|parent| {
            parent.spawn((Ship, Name::new("Ship 1")));
            parent.spawn((Ship, Name::new("Ship 2")));
        });
}
```

我们也可以直接向 `children!` 宏传入一批子实体来替代闭包：

<details markdown="1">
<summary>原文</summary>

Instead of the closure we can pass a bundle of children to the `children!` macro.

</details>

```rust
fn spawn_fleet_with_sugar(mut commands: Commands) {
    commands.spawn((
        Fleet,
        children![
            (Ship, Name::new("Ship 3")),
            (Ship, Name::new("Ship 4")),
        ]
    ));
}
```

真相的源头是 `Relationship` 组件。这就是我们要添加到**其他实体**上来指定其关系（relationship）的组件。
它必须包含一个我们将要附加到的实体的引用。

<details markdown="1">
<summary>原文</summary>

The source of truth is the `Relationship` component.
This is the component we will be adding to __other entities__ to specify the relationship.
It must contain a reference to the entity we will be attaching ourselves to.

</details>

```rust
#[derive(Component)]
#[relationship(relationship_target = ShipAttachments)]
struct AttachedToShip(pub Entity);
```

`RelationshipTarget` 组件会与所有 `AttachedToShip` 组件自动同步。它必须包含一个实体列表来存储他们。

<details markdown="1">
<summary>原文</summary>

The `RelationshipTarget` is the component that will automatically be kept in sync
with all our `AttachedToShip` components. It must contain a list of entities to store them.

</details>

```rust
#[derive(Component)]
#[relationship_target(relationship = AttachedToShip, linked_spawn)]
struct ShipAttachments(Vec<Entity>);
```

`linked_spawn` 允许我们移除 `ShipAttachments`，同时 Bevy 会自动销毁其他实体上的 `AttachedToShip` 关系组件。

<details markdown="1">
<summary>原文</summary>

The `linked_spawn` will allow us to remove the `ShipAttachments` and
Bevy will automatically despawn any `AttachedToShip` components on our other entities.

</details>

要创建关系，我们可以直接在其他实体上生成 `Relationship` 组件：

<details markdown="1">
<summary>原文</summary>

To create the relationship we can then spawn this `Relationship` on other entities.

</details>

```rust
fn spawn_ship(mut commands: Commands) {
    // Spawn the parent Ship
    let ship = commands.spawn((Ship, Name::new("Ship"))).id();

    // Spawn a GunTurret and attach it to the Ship using the new Relationship
    // component
    commands.spawn((GunTurret, AttachedToShip(ship), Name::new("GunTurret 1")));
    commands.spawn((GunTurret, AttachedToShip(ship), Name::new("GunTurret 2")));
}
```

或者，从父实体侧利用 `related!` 宏来声明关系可以更加简洁：

<details markdown="1">
<summary>原文</summary>

This can be shortened by using the `related!` macro to specify the relationships from the parent entity instead:

</details>

```rust
fn build_ship(mut commands: Commands) {
    // Spawn a Ship entity
    commands.spawn((
        Ship,
        Name::new("Ship A"),
        related!(ShipAttachments[
      // Attach GunTurrets to the Ship using the relationship
      (GunTurret, Name::new("GunTurret 1")),
      (GunTurret, Name::new("GunTurret 2")),
    ]),
    ));
}
```

关系是以组件形式存储的，因此可以直接查询：

<details markdown="1">
<summary>原文</summary>

Relationships are stored as components so we can query them:

</details>

```rust
fn log_ship_report(
    ships: Query<(&Name, &ShipAttachments), With<Ship>>,
    turrets: Query<&Name, With<GunTurret>>,
) {
    for (ship_name, attachments) in &ships {
        info!("{} has the following attachments:", ship_name.as_str());

        for &attachment in &attachments.0 {
            if let Ok(child_name) = turrets.get(attachment) {
                info!(" - {}", child_name.as_str());
            }
        }
    }
}
```

你可以从关系的任一侧遍历其中关联：

<details markdown="1">
<summary>原文</summary>

You can iterate the association from either side of the relationship:

</details>

```rust
fn iterate_from_turrets_to_ships(
    ships: Query<Entity, With<Ship>>,
    turrets: Query<Entity, With<AttachedToShip>>,
    attachments: Query<&AttachedToShip>,
) {
    for turret in &turrets {
        for attached in attachments.iter_ancestors(turret) {
            let ship = ships.get(attached);

            info!("Turret {:?} is attached to Ship {:?}", turret, ship);
        }
    }
}

fn iterate_from_ships_to_turrets(
    ships: Query<Entity, With<Ship>>,
    turrets: Query<Entity, With<GunTurret>>,
    ship_attachments: Query<&ShipAttachments>,
) {
    for ship in &ships {
        for attachment in ship_attachments.iter_descendants(ship) {
            let turret_entity = turrets.get(attachment);
            info!("Ship {:?} has Turret {:?}", ship, turret_entity);
        }
    }
}
```

如果关系中包含环路，切记不要这么做，否则会导致无限循环。

<details markdown="1">
<summary>原文</summary>

You must be careful not to use this if your relationships contain loops as this will run infinitely.

</details>

Bevy 目前还没有原生的多对多关系表达方式，`ChildOf` 只能指向单个实体。

<details markdown="1">
<summary>原文</summary>

Bevy does not currently have a native way of representing many-to-many relationships.
`ChildOf` can only point to a single entity.

</details>

## 输入（Input）

在 Bevy 中处理输入有两种方式：

1. 响应 Bevy 输入系统自动发出的事件
2. 查询诸如 `ButtonInput`、`Axis`、`Touches` 或 `Gamepads` 之类的资源

<details markdown="1">
<summary>原文</summary>

There are two ways to handle input in Bevy:

1. Reacting to the events emitted automatically by Bevy's input systems
2. Querying a resource like `ButtonInput`, `Axis`, `Touches` or `Gamepads`

</details>

Bevy 为每一种输入类型都提供了不同的资源：

| 资源              | 描述                 |
|-----------------|--------------------|
| `Axis`          | 存储特定输入设备的位置数据      |
| `ButtonInput`   | 一个“可按压的”输入         |
| `GamepadAxis`   | 一个游戏手柄的一个轴         |
| `GamepadButton` | 如键盘一样表示一个游戏手柄的单个按钮 |
| `Gamepads`      | 表示已连接的游戏控制器的一个集合   |
| `TouchInput`    | 表示触控输入事件           |
| `Touches`       | 已经发生的 `Touch` 事件集合 |

<details markdown="1">
<summary>原文</summary>

Bevy has a different resource for each type of input:

| Resource        | Description                                                  |
|-----------------|--------------------------------------------------------------|
| `Axis`          | stores the position data from certain input devices          |
| `ButtonInput`   | a "press-able" input                                         |
| `GamepadAxis`   | An axis of a gamepad                                         |
| `GamepadButton` | represents a single button of a gamepad just like a keyboard |
| `Gamepads`      | represents a collection of connected game controllers        |

每个 `Camera` 主要负责三件事：

<details markdown="1">

1. The __render target__ which is the region of the screen to draw something

- X increases going to the right
- Y increases going up
- Z increases coming towards the screen
- The default center of the screen is (0, 0)

```rust
#[require(Camera2d)]

    commands.spawn(MainCamera);
}

fn move_camera(
    mut camera: Single<&mut Transform, With<MainCamera>>,
    player: Single<&Transform, With<Player>>,
    time: Res<Time>,
) {
    let direction = Vec3::new(
        player.translation.x,
        player.translation.y,
        camera.translation.z,
    );

    camera.translation =
        camera.translation.lerp(direction, time.delta_secs() * 2.);
}

fn rotate_camera_to_mouse(
    time: Res<Time>,
    mut mouse_motion: MessageReader<MouseMotion>,
    mut transform: Single<&mut Transform, With<Camera>>,
) {
    let dt = time.delta_secs();
    // The factors are just arbitrary mouse sensitivity values.
    // It's often nicer to have a faster horizontal sensitivity than vertical.
    let mouse_sensitivity = Vec2::new(0.12, 0.10);

    for motion in mouse_motion.read() {
        let delta_yaw = -motion.delta.x * dt * mouse_sensitivity.x;
        let delta_pitch = -motion.delta.y * dt * mouse_sensitivity.y;

        // Add yaw which is turning left/right (global)
        transform.rotate_y(delta_yaw);

        // Add pitch which is looking up/down (local)
        const PITCH_LIMIT: f32 = std::f32::consts::FRAC_PI_2 - 0.01;
        let (yaw, pitch, roll) = transform.rotation.to_euler(EulerRot::YXZ);
        let pitch = (pitch + delta_pitch).clamp(-PITCH_LIMIT, PITCH_LIMIT);

        // Apply the rotation
        transform.rotation = Quat::from_euler(EulerRot::YXZ, yaw, pitch, roll);
    }
}
```

## 拾取（Picking）

要启用点击、拖拽和其他交互，我们使用 Bevy 内置的拾取（picking）库。

<details markdown="1">
<summary>原文</summary>

To enable clicking, dragging and other interactions we use bevy's built in picking library.

</details>

拾取是以 `PointerLocation` 组件作为后端，并生成包含这些指针及其所命中实体的 `PointerHits` 事件。

<details markdown="1">
<summary>原文</summary>

A picking backend `PointerLocation` components and produce `PointerHits` events
that contain these pointers and the entities they are hitting.

</details>

一个应用可以同时启用多个拾取后端。

<details markdown="1">
<summary>原文</summary>

An app can have multiple picking backends active at once.

</details>

一个 `PointerHit` 事件中包含了一个指针当前命中的实体信息：

<details markdown="1">
<summary>原文</summary>

A `PointerHit` event contains information about what entities a pointer is currently hitting:

</details>

```rust
pub struct PointerHits {
    pub pointer: PointerId,
    pub picks: Vec<(Entity, HitData)>,
    pub order: f32,
}
```

这些 `PointerHits` 随后会被通用拾取插件用来生成更高层级的 `Pointer<E>` 事件，
然后我们可以通过 `Observer` 或 `MessageReader` 来响应。

<details markdown="1">
<summary>原文</summary>

These `PointerHits` are then used by the generic picking plugins to produce higher level `Pointer<E>` events
which we can react to with either an `Observer` or `MessageReader`.

</details>

一个 `Pointer` 事件是 Bevy 用来为指针事件保存公共数据的类型。它们可以来自三类对象：

1. 窗口
2. 图像
3. GPU 纹理视图

<details markdown="1">
<summary>原文</summary>

A `Pointer` event is what Bevy uses to store the common data needed for pointer events.
They can come from 3 things:

1. Windows
2. Images
3. GPU Texture Views

</details>

如果启用了 `bevy_picking` 特性，Bevy 的 `DefaultPlugins` 会包含 `DefaultPickingPlugins`，其中包括：

| Plugin               | 描述             |
|----------------------|----------------|
| `InteractionPlugin`  | 生成指针事件并处理事件冒泡  |
| `PickingPlugin`      | 管理拾取状态并生成更高层事件 |
| `PointerInputPlugin` | 用于拾取指针的鼠标和触摸事件 |

<details markdown="1">
<summary>原文</summary>

Bevy's `DefaultPlugins` includes the `DefaultPickingPlugins` if you have
the `bevy_picking` feature enabled which contains:

| Plugin               | Description                                                 |
|----------------------|-------------------------------------------------------------|
| `InteractionPlugin`  | Generates pointer events and handles event bubbling         |
| `PickingPlugin`      | Manages the picking state and generates higher level events |
| `PointerInputPlugin` | Mouse and touch events for picking pointers                 |

</details>

可以通过添加 `Pickable::IGNORE` 来禁用某个实体的拾取功能。

<details markdown="1">
<summary>原文</summary>

Picking can be disabled for individual entities by adding `Pickable::IGNORE`.

</details>

精灵图和 UI 元素已经默认启用了拾取后端。要让网格也支持拾取，需要添加 `MeshPickingPlugin`。

<details markdown="1">
<summary>原文</summary>

Sprites and UI elements have a picking backend already enabled.
To enable picking meshes we need to add the `MeshPickingPlugin`.

</details>

在单帧内，拾取事件的顺序如下：

1. `Out` -> `DragLeave`
2. `DragEnger` -> `Over`

<details markdown="1">
<summary>原文</summary>

Within a single frame, the order of picking events is:

1. `Out` -> `DragLeave`
2. `DragEnger` -> `Over`
</details>

然后以下事件都可能以任意顺序发生：

- 每次移动：`DragStart` -> `Drag` -> `DragOver` -> `Move`
- 每次按钮按下：`Press` 或 `Click` -> `Release` -> `DragDrop` -> `DragEnd` -> `DragLeave`
- 每次指针取消：`Cancel`

<details markdown="1">
<summary>原文</summary>

Then any of the following can happen in any order:

- For each movement: `DragStart` -> `Drag` -> `DragOver` -> `Move`
- For each button press: `Press` or `Click` -> `Release` -> `DragDrop` -> `DragEnd` -> `DragLeave`
- For each pointer cancellation: `Cancel`

</details>

由交互状态机来管理跨帧：

- 指针移动到目标上时：`Over` -> `Move` -> `Out`
- 指针在目标上按下按钮时：`Press` -> `Click` -> `Release`
- 指针拖拽目标时：`DragStart` -> `Drag` -> `DragEnd`
- 指针在目标上拖拽物体时：`DragEnter` -> `DragOver` -> `DragDrop` -> `DragLeave`
- 指针被取消时：该指针后续不会再有其他事件跟随 `Cancel`

<details markdown="1">
<summary>原文</summary>

Between frames things are managed by the interaction state machine:

- When a pointer moves over the target: `Over` -> `Move` -> `Out`
- When a pointer presses buttons on the target: `Press` -> `Click` -> `Release`
- When a pointer drags the target: `DragStart` -> `Drag` -> `DragEnd`
- When a pointer drags something over the target: `DragEnter` -> `DragOver` -> `DragDrop` -> `DragLeave`
- When a pointer is canceled: No other events will follow the `Cancel` event for that pointer.

</details>

## 计时器（Timers）

计时器有两种模式：

1. `TimerMode::Once`：计时到 0 一次，需要手动重置
2. `TimerMode::Repeat`：计时到 0 然后会自动重置

<details markdown="1">
<summary>原文</summary>

Timers come in two modes:

1. `TimerMode::Once` which will tick down to 0 once, and only resets manually
2. `TimerMode::Repeat` which will tick down to 0 then reset itself automatically

</details>

在 Bevy 中，计时器不是从初始值往下减，而是从 0 往上累计，直到达到它的 `Duration`。

<details markdown="1">
<summary>原文</summary>

In Bevy, timers don't tick down from their initial value.
Instead they tick up from zero until they reach their `Duration`.

</details>

当计时结束，之后我们可以调用 `finished` 或 `just_finished` 来切换行为。
两者的区别在于，`just_finished` 只有在计时器在上一帧中刚好结束时才返回 `true`。

<details markdown="1">
<summary>原文</summary>

We can then call `finished` or `just_finished` to switch behavior when they are done.
The difference between them is that `just_finished` only returns `true` if the timer finished in the last tick.

</details>

```rust
#[derive(Resource, Default)]
pub struct MatchTime(Timer);

impl MatchTime {
    pub fn new() -> Self {
        Self(Timer::from_seconds(60.0, TimerMode::Once))
    }
}

fn countdown(time: Res<Time>, mut match_time: ResMut<MatchTime>) {
    match_time.0.tick(time.delta());
}

fn end_match(match_time: Res<MatchTime>) {
    if match_time.0.is_finished() {
        // Here we would rest our game
    }
}
```

Bevy 内置了 `Time` 资源，我们可以用它获取当前帧与上一帧之间经过的秒数。

<details markdown="1">
<summary>原文</summary>

Bevy has a built in `Time` resource we can use to get the `delta` in seconds of
the time between this tick and the last.

</details>

```rust

    info!("Duration passed: {:?}", time.delta());
    info!("Seconds passed: {:?}", time.delta_secs_f64());
    info!("Total time since startup: {:?}", time.elapsed());
}
```

假设我们想让某个技能拥有一个 `Cooldown`。这就不适合作为 `Resource`，因为冷却时间是某个玩家专属的。

<details markdown="1">
<summary>原文</summary>

Now lets say we wanted to have a `Cooldown` for one of our abilities.
This wouldn't make sense as a `Resource` because the cooldown would be specific to one of our players.

</details>

```rust
#[derive(Component)]
struct Cooldown(Timer);

#[derive(Component)]
struct Player;

fn cast_spell(
    mut commands: Commands,
    mut player_query: Query<Entity, With<Player>>,
    cooldowns: Query<&Cooldown, With<Player>>,
) {
    if let Ok(player) = player_query.single() {
        if let Ok(cooldown) = cooldowns.get(player) {
            info!(
        "You cannot cast yet. Your cooldown is {:0.0}% complete!",
        cooldown.0.fraction() * 100.0
      )
        } else {
            // Add an entity to the world with a timer
            commands
                .entity(player)
                .insert(Cooldown(Timer::from_seconds(5.0, TimerMode::Once)));

            // Cast the spell here
        }
    }
}

fn tick_cooldowns(
    mut commands: Commands,
    mut cooldowns: Query<(Entity, &mut Cooldown)>,
    time: Res<Time>,
) {
    for (entity, mut cooldown) in &mut cooldowns {
        cooldown.0.tick(time.delta());

        if cooldown.0.is_finished() {
            commands.entity(entity).remove::<Cooldown>();
        }
    }
}
```

有时你会希望某个计时器只在单个系统中使用。对于这种情况，`Resource` 会显得太公开，你可能会更倾向于使用 `Local` 系统参数。

<details markdown="1">
<summary>原文</summary>

Sometimes you will want a timer that is only ever used in a single system.
For these cases a `Resource` would be too public and you might prefer a `Local` system parameter instead:

</details>

```rust

    timer.tick(time.delta());

    if timer.just_finished() {
        info!("The timer is finished");
    }
}
```

<summary>原文</summary>

The data must be one of the file formats supported by Bevy:

</details>

- `wav`
- `ogg`
- `flac`
- `mp3`

接收器（sink）是声音数据的目的地。这是源数据将要发送的地方，也是之后被发送到全局侦听器的地方。

<details markdown="1">
<summary>原文</summary>

A sink is a destination for the sound data.
This is the place where sources will send their data and will be emitted to the global listener.

</details>

```rust
fn play_pitch(
    mut pitch_assets: ResMut<Assets<Pitch>>,
    mut commands: Commands,
) {
    info!("playing pitch with frequency: {}", 220.0);
    commands.spawn((
        AudioPlayer(pitch_assets.add(Pitch::new(220.0, Duration::new(1, 0)))),
        PlaybackSettings::DESPAWN,
    ));
}
```

有几种不同的内置播放设置：

| 设置                          | 描述            |
|-----------------------------|---------------|
| `PlaybackSettings::ONCE`    | 只会播放一次关联音频    |
| `PlaybackSettings::LOOP`    | 会循环音频         |
| `PlaybackSettings::DESPAWN` | 会播放一次音频然后销毁实体 |
| `PlaybackSettings::REMOVE`  | 会播放一次音频然后循环组件 |

<details markdown="1">
<summary>原文</summary>

There are a few different playback settings that are built in:

| Setting                     | Description                                         |
|-----------------------------|-----------------------------------------------------|
| `PlaybackSettings::ONCE`    | Will play the associated audio only once            |
| `PlaybackSettings::LOOP`    | Will loop the audio                                 |
| `PlaybackSettings::DESPAWN` | Will play the audio once then despawn the entity    |
| `PlaybackSettings::REMOVE`  | Will play the audio once then despawn the component |

</details>

我们可以通过在任意实体上生成 `AudioPlayer` 来触发声音播放。

<details markdown="1">
<summary>原文</summary>

We can trigger our sounds to play by spawning an `AudioPlayer` on any entity.

</details>

```rust
fn play_background_audio(
    asset_server: Res<AssetServer>,
    mut commands: Commands,
) {
    let audio = asset_server.load("background_audio.ogg");

    // Create an entity dedicated to playing our background music
    commands.spawn((
        AudioPlayer::new(audio),
        PlaybackSettings::LOOP,
    ));

    // Spawn our listener
    commands.spawn((
        SpatialListener::new(100.), // Gap between the ears
        Transform::default(),
    ));
}
```

一旦资源加载完成，音乐会开始循环播放，直到我们创建的实体被销毁或组件被移除。

<details markdown="1">
<summary>原文</summary>

Once the asset is loaded the music will start playing in a loop until
this entity we spawned is despawned or the component is removed.

</details>

要控制 `AudioPlayer` 的播放状态，我们可以使用在创建实体时被 `AudioPlugin` 自动添加 `AudioSink` 组件：

<details markdown="1">
<summary>原文</summary>

To control the playback of our `AudioPlayer` we can use the `AudioSink`
which was added by the `AudioPlugin` automatically when we spawned our entity:

</details>

```rust
fn pause(
    keyboard_input: Res<ButtonInput<KeyCode>>,
    music_controller: Query<&AudioSink, With<MusicBox>>,
) {
    let Ok(sink) = music_controller.single() else {
        return;
    };

    if keyboard_input.just_pressed(KeyCode::Space) {
        sink.toggle_playback();
    }
}
```

`AudioSink` 是我们的公共 API，支持：

| 方法                | 描述                 |
|-------------------|--------------------|
| `play`            | 恢复播放               |
| `pause`           | 暂停播放               |
| `stop`            | 终止播放，之后不能再重新开始     |
| `mute`            | 静音播放               |
| `unmute`          | 解除静音播放             |
| `toggle_playback` | 切换播放状态             |
| `toggle_mute`     | 切换静音状态             |
| `is_paused`       | 暂停时返回 `true`       |
| `is_muted`        | 静音时返回 `true`       |
| `speed`           | 获取播放速度             |
| `set_speed`       | 控制播放速度             |
| `empty`           | 没有更多声音播放时返回 `true` |
| `try_seek`        | 在源声音中搜索特定位置        |

<details markdown="1">
<summary>原文</summary>

The `AudioSink` is our public API to:

| Method            | Description                                         |
|-------------------|-----------------------------------------------------|
| `play`            | Resumes playback                                    |
| `pause`           | Pause playback                                      |
| `stop`            | Stop the playback, cannot be restarted after        |
| `mute`            | Mute the playback                                   |
| `unmute`          | Unmute the playback                                 |
| `toggle_playback` | Toggle the playback                                 |
| `toggle_mute`     | Toggle muting the playback                          |
| `is_paused`       | Returns true if the sink is paused                  |
| `is_muted`        | Returns true if the sink is muted                   |
| `speed`           | Get the speed of the sound                          |
| `set_speed`       | Control the speed of the playback                   |
| `empty`           | Returns true if the sink has no more sounds to play |
| `try_seek`        | Seek to a certain point in the source sound         |

</details>

There are two separate sources of volume for our apps:

## 场景（Scenes）

场景可以序列化成基于文件的表示形式。所有内容都会被序列化到文件中，然后在场景加载时重新初始化。

<details markdown="1">
<summary>原文</summary>

Scenes can be serialized into file based representations.
Everything is serialized into a file and then reinitialized when the scene is loaded.

</details>

场景会保存为 `.scn` 或 `.scn.ron` 文件。其文件格式基于 [Rusty Object Notation (RON)](https://crates.io/crates/ron)。

<details markdown="1">
<summary>原文</summary>

Scenes are saved into a `.scn` or `.scn.ron`. The format of the file is based on
[Rusty Object Notation (RON)](https://crates.io/crates/ron).

</details>

我们可以使用 `DynamicScene::serialize` 方法将场景写入文件：

<details markdown="1">
<summary>原文</summary>

We save scenes to a file by using the `DynamicScene::serialize` method:

</details>

```rust
fn save_scene_system(world: &mut World) {
    let scene = DynamicScene::from_world(world);

    // Scenes can be serialized like this:
    let type_registry = world.resource::<AppTypeRegistry>();
    let type_registry = type_registry.read();
    let serialized_scene = scene.serialize(&type_registry).unwrap();

    // Showing the scene in the console
    info!("{}", serialized_scene);

    // Writing the scene to a new file. Using a task to avoid calling the
    // filesystem APIs in a system as they are blocking This can't work in WASM as
    // there is no filesystem access
    #[cfg(not(target_arch = "wasm32"))]
    IoTaskPool::get()
        .spawn(async move {
            // Write the scene RON data to file
            File::create(format!("assets/{NEW_SCENE_FILE_PATH}"))
                .and_then(|mut file| file.write(serialized_scene.as_bytes()))
                .expect("Error while writing scene to file");
        })
        .detach();
}
```

当 Bevy 加载场景文件时，需要把它反序列化成真正的组件和实体，然后加载到你的世界中。

<details markdown="1">
<summary>原文</summary>

When Bevy loads the scene file, it needs to deserialize it into actual components and entities
that it loads into your world.

</details>

生成场景有 3 种方式：

1. 使用 `SceneSpawner::spawn_dynamic`
2. 给实体添加 `DynamicSceneRoot` 组件
3. 使用 `DynamicSceneBuilder` 从一个 `World` 构造 `DynamicScene`

<details markdown="1">
<summary>原文</summary>

There are 3 ways to spawn scenes:

1. Using `SceneSpawner::spawn_dynamic`
2. Adding the `DynamicSceneRoot` component to an entity
3. Using the `DynamicSceneBuilder` to construct a `DynamicScene` from a `World`

</details>

最简单的是直接生成 `DynamicSceneRoot`，它会使用 `SceneLoader` 反序列化所有内容：

<details markdown="1">
<summary>原文</summary>

The easiest of these is simply spawning a `DynamicSceneRoot`.
It uses the `SceneLoader` to deserialize everything:

</details>

```rust
const SCENE_FILE_PATH: &str = "scene.ron";

fn load_scene_system(mut commands: Commands, asset_server: Res<AssetServer>) {
    // "Spawning" a scene bundle creates a new entity and spawns new instances
    // of the given scene's entities as children of that entity.
    let scene = asset_server.load(SCENE_FILE_PATH);
    commands.spawn(DynamicSceneRoot(scene));
}
```

一旦场景加载完成，`SceneInstance` 组件会被添加到对应实体，它可以和 `SceneSpawner` 一起用来操作该场景。

<details markdown="1">
<summary>原文</summary>

Once the scene has been loaded, a `SceneInstance` component is added to the component
which can be used with the `SceneSpawner` to interact with the scene.

</details>

例如，我们可以销毁所有已加载的场景：

<details markdown="1">
<summary>原文</summary>

For example we could despawn all our loaded scenes:

</details>

```rust
use bevy::scene::SceneInstance;

fn despawn_all_scenes(
    query: Query<&SceneInstance>,
    mut spawner: ResMut<SceneSpawner>,
    world: &mut World,
) {
    // Despawning the scene root entity will also despawn all of its children
    for instance in &query {
        spawner.despawn_instance_sync(world, instance);
    }
}
```

`FromWorld` 特征决定了组件在加载进 `World` 时如何构造。

<details markdown="1">
<summary>原文</summary>

The `FromWorld` trait determines how your component is constructed when it loads into the `World`.

</details>

在组件上实现 `FromWorld`，可以让你用当前 `World` 的资源来自定义初始化：

<details markdown="1">
<summary>原文</summary>

Implementing `FromWorld` on a component will let you customize initialization
using the current Worlds resources:

</details>

```rust
impl FromWorld for ComponentB {
    fn from_world(world: &mut World) -> Self {
        let time = world.resource::<Time>();
        ComponentB {
            _time_since_startup: time.elapsed(),
            value: "Default Value".to_string(),
        }
    }
}
```

## 物理（Physics）

Bevy 没有内置物理引擎。最接近 Bevy 风格的选择是 `avian`。

<details markdown="1">
<summary>原文</summary>

Bevy does not have a built-in physics engine. The most native to Bevy is `avian`.

</details>

从 Bevy 渲染器的视角来看，你的位置由实体的 `Transform` 组件决定。

<details markdown="1">
<summary>原文</summary>

Your position, in the eyes of Bevy's renderer, is dictated by an entity's `Transform` component.

</details>

在 Avian 中，我们可以使用更方便一些的 `Position` 组件。它会由 Avian 的 `SyncPlugin` 自动与 `Transform` 保持同步。

<details markdown="1">
<summary>原文</summary>

In Avian we can use a somewhat more convenient `Position` component.
This is kept in sync with the `Transform` automatically by Avian's `SyncPlugin`.

</details>

```rust
fn move_things_with_position(mut query: Query<&mut Position>) {
    for mut position in &mut query {
        position.x += 1.;
    }
}
```

就和 `Position` 类似，Avian 也提供了 `Rotation` 组件。

<details markdown="1">
<summary>原文</summary>

Just like a `Position` Avian provides a `Rotation` component.

</details>

```rust
fn rotate_things(mut query: Query<&mut Rotation>) {
    for mut rotation in &mut query {
        *rotation = rotation.add_angle_fast(0.1);
    }
}
```

刚体有 3 种不同的组件，每一种都适用于不同场景：

1. `RigidBody::Dynamic` 类似现实世界中的物体，会受到力和接触影响
2. `RigidBody::Kinematic` 只能通过程序移动，适合玩家角色控制器和移动平台
3. `RigidBody::Static` 不能移动，适合地面和墙壁这类环境物体

<details markdown="1">
<summary>原文</summary>

Rigid bodies come in 3 different components, each specialized for something:

1. `RigidBody::Dynamic` are similar to real life objects and are affected by forces and contacts.
2. `RigidBody::Kinematic` can only be moved programmatically, which is useful for things like player character controllers and moving platforms.
3. `RigidBody::Static` can not move, so they can be good for objects in the environment like the ground and walls.

</details>

要移动物体，我们可以直接控制 `Position`，也可以使用 `LinearVelocity` 组件：

<details markdown="1">
<summary>原文</summary>

To move things we can control the `Position` directly or use a `LinearVelocity` component:

</details>

```rust
fn spawn_ball(mut commands: Commands) {
    commands.spawn((
        RigidBody::Dynamic,
        LinearVelocity(Vec2::new(0.0, 0.0)),
        Collider::circle(0.5),
        Mass(5.0),
        CenterOfMass::new(0.0, -0.5),
    ));
}
```

Avian 会使用这个碰撞体的大小来决定你的刚体有多少质量。

<details markdown="1">
<summary>原文</summary>

Avian is going to use the size of this collider to determine how much mass your
body has.

</details>

添加 `Sensor` 组件能让你检测碰撞，而不会影响实体的质量属性，也不会与其他物理体发生相互作用。

<details markdown="1">
<summary>原文</summary>

Adding a `Sensor` component will let you detect collisions without affecting the
entity's mass properties or interacting with other physical bodies.

</details>

```rust
fn spawn_sensor(mut commands: Commands) {
    commands.spawn((
        RigidBody::Dynamic,
        Collider::circle(0.5),
        Sensor,
    ));
}
```

如果要一次处理大量碰撞，就应该使用 `MessageReader`：

<details markdown="1">
<summary>原文</summary>

For processing a large number of collisions at once you would use the `MessageReader`:

</details>

```rust
fn react_to_collisions(mut collision_events: MessageReader<CollisionStart>) {
    for event in collision_events.read() {
        info!(
            "Collision started between {:?} and {:?}",
            event.collider1,
            event.collider2
        );
    }
}
```

但如果我们想要实体级别的碰撞处理，就可以使用观察器：

<details markdown="1">
<summary>原文</summary>

However if we want entity-specific collisions then we can use observers:

</details>

```rust
#[derive(Component)]
struct SecurityCamera;

#[derive(Component)]
struct Enemy;

fn setup_security_cameras(mut commands: Commands) {
    commands
        .spawn((
            SecurityCamera,
            Collider::circle(3.0), // Detection radius
            Sensor,
            CollisionEventsEnabled, // So we receive collision events
        ))
        .observe(|trigger: On<CollisionStart>, enemy_query: Query<&Enemy>| {
            let camera = trigger.collider1;
            let intruder = trigger.collider2;
            if enemy_query.contains(intruder) {
                println!("Security camera {camera} detected enemy {intruder}!");
            }
        });
}
```
